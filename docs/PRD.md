# Invoice MVP PRD 메타 프롬프트

이 문서는 Invoice MVP 또는 유사한 SaaS 프로젝트의 PRD(Product Requirements Document)를 작성하기 위한 메타 프롬프트입니다. 에이전트 또는 개발자가 이 템플릿을 따라 일관성 있는 PRD를 생성할 수 있습니다.

---

## 이 문서의 사용 방법

1. 아래 "프롬프트 지시문"을 AI 에이전트(Claude, GPT 등)에 그대로 붙여 넣습니다.
2. `[대괄호]` 안의 내용을 실제 프로젝트 정보로 교체합니다.
3. 에이전트가 생성한 PRD를 "정합성 검증 체크리스트"로 검토합니다.
4. 실패 항목이 있으면 해당 섹션을 수정하고 체크리스트를 재실행합니다.

---

```
- 프로젝트명: [Invoice MVP]
- 핵심 문제: [Notion에 입력한 견적서 데이터를 클라이언트가 웹에서 확인하고 PDF로 저장할 수 없음]
- 사용자 유형: [관리자(프리랜서/사업자), 클라이언트(견적서 수신자)]
- 기술 스택: [Next.js 16, React 19, TypeScript, Supabase, Notion API]
- 외부 연동: [Notion API - 데이터 소스, Supabase - 인증 및 DB]

## 절대 생성하지 말 것

- 개발 우선순위, 성능 지표, API 라우트 목록
- 인프라 구성도, 마일스톤, 스프린트 계획
- 개발 단계(Phase 1/2/3), 개발 워크플로우
- 보안 요구사항 명세, 사용자 페르소나 문서

## 반드시 포함할 것

아래 7개 섹션을 순서대로 작성하세요.

### 섹션 1. 핵심 정보 (2줄)
- 목적: 해결하는 핵심 문제를 1줄로
- 사용자: 구체적인 사용자층을 1줄로 (모든 사용자 유형 포함)

### 섹션 2. 사용자 여정
- 전체 플로우를 번호 목록으로 (페이지 이름만, URL 경로 제외)
- 각 단계마다 조건 분기점 명시 (예: [로그인됨] / [비로그인])
- 자동 리디렉션 조건 명시

### 섹션 3. 기능 명세
MVP 핵심 기능과 MVP 필수 지원 기능으로 나누어 테이블로 작성합니다.

테이블 컬럼: ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지

규칙:
- 각 기능에 F001, F002... 형식의 ID 부여
- 관련 페이지는 페이지 이름만 (URL 경로 없음)
- MVP 핵심 기능: 서비스 고유 가치를 제공하는 기능만 (F001~F009)
- MVP 필수 지원 기능: 인증 등 서비스 운영에 필수적인 공통 기능 (F010~)
- MVP 이후 기능: 제외할 기능 목록을 글머리 기호로 나열

### 섹션 4. 메뉴 구조
트리 형태로 전체 내비게이션을 표현합니다.

규칙:
- 접근 권한별로 구분 (공개 영역 / 비로그인 메뉴 / 로그인 후 메뉴)
- 각 메뉴 항목 옆에 관련 기능 ID 명시 (예: 대시보드 - F001, F002)
- URL 경로 없이 메뉴 이름만 사용
- 모든 메뉴 항목은 페이지별 상세 기능에 해당 페이지가 반드시 존재해야 함

### 섹션 5. 페이지별 상세 기능
각 페이지를 아래 5개 항목의 테이블로 작성합니다.

항목: 역할 | 진입 경로 | 사용자 행동 | 주요 기능 | 다음 이동

규칙:
- 페이지 제목 아래 "구현 기능: F001, F002" 형식으로 기능 ID 명시
- 주요 기능은 구체적으로 (예: "UUID 기반 공개 토큰 자동 생성" O, "토큰 생성" X)
- 다음 이동은 성공/실패/조건별로 모두 명시
- 메뉴 구조의 모든 항목이 이 섹션에 페이지로 존재해야 함

### 섹션 6. 데이터 모델
각 모델을 테이블로 작성합니다.

테이블 컬럼: 필드 | 설명 | 타입/관계

규칙:
- 핵심 필드 5개 내외
- 외래 키는 "→ [모델명].id" 형식으로 관계 표현
- 기능 명세의 핵심 데이터를 모두 커버해야 함

### 섹션 7. 기술 스택
카테고리별로 나누어 작성합니다.

카테고리: 프론트엔드 프레임워크 | 스타일링 & UI | 폼 & 검증 | 외부 연동 | 백엔드 & DB | 배포

규칙:
- 현재 프로젝트에 실제 사용 중인 버전 명시
- 각 기술의 선택 이유를 짧게 설명
- 외부 연동(Notion API 등)은 별도 카테고리로 분리

## 문서 정합성 원칙 (작성 중 반드시 준수)

1. 기능 명세의 모든 기능 ID는 페이지별 상세 기능에서 반드시 구현되어야 함
2. 페이지별 상세 기능의 모든 기능 ID는 기능 명세에 정의되어 있어야 함
3. 메뉴 구조의 모든 항목은 페이지별 상세 기능에 해당 페이지가 있어야 함
4. 한 섹션에만 존재하는 기능이나 페이지는 허용하지 않음

위 지시문에 따라 Invoice MVP PRD를 작성해주세요.
```

## PRD 섹션별 작성 가이드

### 섹션 1. 핵심 정보

**작성 목적**: 문서를 읽는 사람이 5초 안에 프로젝트의 본질을 파악하게 합니다.

**작성 가이드**:
- 목적은 "~을 통해 ~을 해결한다" 구조로 작성합니다.
- 사용자는 서비스를 사용하는 모든 역할을 나열합니다.

**좋은 예시**:
```
목적: Notion에 입력한 견적서 데이터를 클라이언트가 고유 링크로 웹에서 확인하고 PDF로
      다운로드할 수 있게 하여, 이메일 첨부 방식의 비효율을 제거한다.
사용자: 프리랜서/소규모 사업자(관리자)와 견적서를 받는 클라이언트(수신자)
```

**나쁜 예시**:
```
목적: 견적서 관리 시스템  (너무 모호)
사용자: 사용자  (구체성 없음)
```

---

### 섹션 2. 사용자 여정

**작성 목적**: 전체 서비스 흐름을 한눈에 보여줍니다. 개발자가 페이지 간 이동 로직을 구현할 때 참고합니다.

**작성 가이드**:
- 번호 목록으로 단계를 표현합니다.
- 각 단계에서 발생하는 분기를 들여쓰기로 표현합니다.
- 자동 리디렉션이 발생하는 조건을 명시합니다.
- 페이지 이름만 사용하고 URL 경로(예: `/dashboard`)는 절대 쓰지 않습니다.

**좋은 예시**:
```
1. 관리자가 Notion에 견적서 입력 후 대시보드 접근
   ↓ 로그인 여부 확인

   [비로그인] → 로그인 페이지 → 로그인 성공 → 대시보드 페이지
   [로그인됨] → 대시보드 페이지 (바로 진입)
```

**자주 하는 실수**:
- 분기 조건 없이 선형으로만 작성 (예외 상황 누락)
- URL 경로를 페이지 이름 대신 사용
- 외부 시스템(Notion) 연동 단계 누락

---

### 섹션 3. 기능 명세

**작성 목적**: 개발해야 할 기능의 목록과 우선순위 기준을 정의합니다. 페이지별 상세 기능의 기준점 역할을 합니다.

**작성 가이드**:

MVP 핵심 기능 선별 기준:
- 이 기능 없이 서비스 고유 가치를 제공할 수 없는가?
- 경쟁 서비스와 차별화되는 기능인가?

MVP 필수 지원 기능 선별 기준:
- 핵심 기능을 사용하기 위해 반드시 필요한가?
- 인증, 기본 데이터 관리 등 공통 인프라에 해당하는가?

**MVP에서 제외해야 할 기능 판단 기준**:
- 첫 사용자가 서비스 가치를 경험하지 않아도 되는 기능인가?
- 나중에 추가해도 핵심 가치가 훼손되지 않는 기능인가?

**좋은 예시 (F001)**:
```
| F001 | Notion 데이터 동기화 | Notion API로 견적서 DB 조회 및 Supabase에 캐싱 |
       데이터 소스 연결 없이 서비스 불가 | 대시보드 페이지, 견적서 공개 페이지 |
```

**자주 하는 실수**:
- 기능 ID 없이 기능명만 나열
- 관련 페이지에 URL 경로 사용
- 하나의 기능이 너무 많은 페이지에 연결됨 (범위 모호)
- MVP 이후 기능을 MVP에 포함

---

### 섹션 4. 메뉴 구조

**작성 목적**: 사용자 접근 권한별 내비게이션 구조를 보여줍니다. 개발자가 미들웨어와 라우트 보호 로직을 구현할 때 참고합니다.

**작성 가이드**:
- 접근 권한 레벨로 그룹을 나눕니다 (공개/비로그인/로그인/관리자 등).
- 각 메뉴 항목에 관련 기능 ID를 명시합니다.
- URL 경로 없이 메뉴 이름만 사용합니다.
- 모든 메뉴 항목은 섹션 5의 페이지로 반드시 연결되어야 합니다.

**Invoice MVP 예시**:
```
공개 영역 (비로그인 접근 가능)
├── 견적서 공개 페이지 (고유 토큰으로 접근)
│   └── 기능: F001, F004, F005
└── 오류 페이지

인증 메뉴 (비로그인 시)
├── 로그인 - F010
└── 회원가입 - F010

관리자 메뉴 (로그인 후)
├── 대시보드 - F001, F002, F003, F011
└── 로그아웃
```

**자주 하는 실수**:
- 메뉴에 페이지가 없는 항목 포함 (섹션 5와 불일치)
- 기능 ID 없이 메뉴명만 나열
- 접근 권한 레벨 구분 없이 모든 메뉴를 하나로 작성

---

### 섹션 5. 페이지별 상세 기능

**작성 목적**: 각 페이지에서 개발자가 구현해야 할 내용을 명확히 정의합니다. 이 섹션만 보고 바로 코딩을 시작할 수 있어야 합니다.

**작성 가이드**:
- 페이지 제목 아래에 구현 기능 ID를 반드시 명시합니다.
- 주요 기능은 사용자가 보는 UI 요소와 동작을 구체적으로 적습니다.
- 다음 이동은 성공/실패/예외 케이스를 모두 커버합니다.
- 이 섹션에 있는 모든 기능 ID는 섹션 3에 정의되어 있어야 합니다.

**좋은 예시 (대시보드 페이지)**:
```
구현 기능: F001, F002, F003, F011 | 인증: 로그인 필수

역할: 관리자의 견적서 목록 확인, 링크 공유, 상태 관리의 메인 작업 화면
진입 경로: 로그인 성공 후 자동 이동, 또는 로그인 상태에서 직접 접근
사용자 행동: 견적서 목록 확인, Notion 수동 동기화, 공개 링크 복사, 상태 변경
주요 기능:
  - 견적서 목록 테이블 (제목, 클라이언트명, 금액, 상태, 생성일)
  - Notion 동기화 버튼 (Notion API 수동 호출 및 데이터 갱신)
  - 행별 공개 링크 복사 버튼 (클립보드에 토큰 포함 URL 복사)
  - 상태 드롭다운 (초안 / 발송됨 / 확인됨)
다음 이동: 공개 링크 복사 → 클립보드 저장 후 토스트 알림, 미리보기 → 모달 오픈
```

**자주 하는 실수**:
- 주요 기능이 너무 추상적 ("데이터 표시" 대신 "견적서 목록 테이블 (컬럼명 나열)")
- 구현 기능 ID 누락
- 다음 이동에서 실패/오류 케이스 누락
- 섹션 3에 없는 기능 ID 사용

---

### 섹션 6. 데이터 모델

**작성 목적**: 데이터베이스 테이블 구조를 정의합니다. 개발자가 Supabase 스키마를 생성할 때 직접 참고합니다.

**작성 가이드**:
- 각 모델은 서비스의 핵심 개념(Entity)에 해당합니다.
- 필드는 핵심 필드 5개 내외로 제한합니다 (모든 필드를 나열하지 않음).
- 외래 키 관계는 "→ [모델명].id" 형식으로 표현합니다.
- 기능 명세에서 필요한 데이터를 모두 커버하는지 확인합니다.

**Invoice MVP 핵심 모델**:

| 모델 | 역할 | 핵심 필드 |
|------|------|-----------|
| Invoice | 견적서 메인 데이터 | notion_page_id, public_token, status, total_amount |
| InvoiceItem | 견적서 항목 | invoice_id, description, quantity, unit_price |
| User | 관리자 계정 | email (Supabase Auth 연동) |

**자주 하는 실수**:
- 불필요한 필드를 모두 나열 (핵심 필드만 선별해야 함)
- 외래 키 관계 누락
- 기능 명세에 필요한 데이터가 모델에 없음 (예: public_token 누락)

### 섹션 7. 기술 스택

**작성 목적**: 사용할 기술과 버전을 확정합니다. 팀원이나 에이전트가 일관된 기술을 사용하도록 합니다.

**작성 가이드**:
- 현재 프로젝트에 실제 설치된 버전을 명시합니다.
- 외부 연동 서비스(Notion, Slack 등)는 별도 카테고리로 분리합니다.
- 각 기술의 역할을 짧게 설명합니다 (개발자가 왜 이 기술을 쓰는지 알 수 있어야 함).

**Invoice MVP 기술 스택 예시**:
```
프론트엔드 프레임워크
- Next.js 16.2.4 (App Router): 서버 컴포넌트로 Notion API 서버 사이드 호출
- React 19.2.4: 현재 프로젝트 기반
- TypeScript 5.6+ (strict 모드): 타입 안전성

Notion 연동
- @notionhq/client: Notion API 공식 클라이언트 (서버 사이드 전용)
- 환경 변수: NOTION_API_KEY, NOTION_DATABASE_ID

PDF 생성
- @react-pdf/renderer: React 컴포넌트를 PDF로 서버 사이드 렌더링
- 대안: window.print() + 인쇄 CSS (MVP 초기 단계 빠른 구현용)
```

## PRD 작성 프로세스 (단계별 순서)

PRD를 작성할 때는 반드시 아래 순서를 따릅니다. 섹션 순서대로 작성하면 정합성 문제가 발생하기 쉽습니다.

**1단계: 사용자 여정 설계 (섹션 2 초안)**
- 서비스의 전체 흐름을 큰 그림으로 먼저 그립니다.
- 모든 사용자 유형의 경로를 포함합니다.

**2단계: 기능 목록 추출 (섹션 3 초안)**
- 사용자 여정의 각 단계에서 필요한 기능을 추출합니다.
- MVP 필수 여부를 판단하여 핵심/지원/제외로 분류합니다.
- 각 기능에 ID를 부여합니다.

**3단계: 메뉴 구조 설계 (섹션 4)**
- 기능 목록을 바탕으로 사용자가 접근하는 메뉴를 설계합니다.
- 각 메뉴 항목에 기능 ID를 매핑합니다.

**4단계: 페이지별 상세 기능 작성 (섹션 5)**
- 메뉴 구조의 모든 항목에 대응하는 페이지를 작성합니다.
- 각 페이지에 구현 기능 ID를 명시합니다.

**5단계: 나머지 섹션 작성**
- 핵심 정보(섹션 1): 완성된 PRD를 요약하여 작성합니다.
- 데이터 모델(섹션 6): 기능 명세에서 필요한 데이터를 모델화합니다.
- 기술 스택(섹션 7): 확정된 기술과 버전을 기록합니다.

**6단계: 정합성 검증 (아래 체크리스트 실행)**

## 정합성 검증 체크리스트

PRD 작성 완료 후 반드시 아래 체크리스트를 순서대로 실행합니다.

### 1단계: 기능 명세 → 페이지 연결 검증

- [ ] 기능 명세(섹션 3)의 모든 기능 ID가 페이지별 상세 기능(섹션 5)에서 구현되었는가?
- [ ] 기능 명세에서 명시한 관련 페이지 이름이 섹션 5에 실제로 존재하는가?

### 2단계: 메뉴 구조 → 페이지 연결 검증

- [ ] 메뉴 구조(섹션 4)의 모든 항목이 섹션 5에 해당 페이지로 존재하는가?
- [ ] 메뉴에서 참조한 모든 기능 ID가 섹션 3에 정의되어 있는가?

### 3단계: 페이지별 상세 기능 → 역참조 검증

- [ ] 섹션 5의 모든 구현 기능 ID가 섹션 3에 정의되어 있는가?
- [ ] 섹션 5의 모든 페이지가 섹션 4 메뉴 구조에서 접근 가능한가?

### 4단계: 데이터 모델 검증

- [ ] 기능 명세(섹션 3)에서 다루는 핵심 데이터가 섹션 6 모델에 존재하는가?
- [ ] 외래 키 관계가 누락된 모델이 없는가?

### 5단계: 누락 및 고아 항목 최종 점검

- [ ] 섹션 3에만 있고 섹션 5에서 구현되지 않은 기능이 있는가? (있으면 제거 또는 페이지 추가)
- [ ] 섹션 5에만 있고 섹션 3에 정의되지 않은 기능이 있는가? (있으면 섹션 3에 추가)
- [ ] 섹션 4에만 있고 섹션 5에 실제 페이지가 없는 항목이 있는가? (있으면 페이지 추가 또는 메뉴에서 제거)

검증 실패 항목이 있으면 해당 항목을 수정한 후 전체 체크리스트를 다시 실행합니다.

|-----------|--------|-------------|

| 기능 ID 불일치 | 섹션 5에 F007이 있는데 섹션 3에 F007이 없음 | 섹션 3 기능 명세를 정합성의 유일한 기준점으로 사용 |
| 메뉴-페이지 불일치 | 메뉴에 "알림" 항목이 있는데 섹션 5에 알림 페이지 없음 | 메뉴 항목 추가 시 섹션 5에 해당 페이지 동시 추가 |
| MVP 범위 초과 | 프로필 관리, 알림 설정 등을 MVP에 포함 | MVP 필수 이유를 "이 기능 없이 서비스 가치를 제공할 수 없는가?"로 판단 |
| 추상적 기능 설명 | "데이터 관리 기능" | "견적서 목록 테이블 (제목, 클라이언트명, 금액, 상태, 생성일)" |
| 다음 이동 누락 | 성공 케이스만 명시, 실패/오류 케이스 없음 | 성공, 실패, 예외 케이스를 모두 "다음 이동" 항목에 명시 |
| 외부 연동 누락 | Notion API, Slack 등 외부 서비스를 기술 스택에 미포함 | 외부 연동 카테고리 별도 분리 및 환경 변수 명시 |

## 유사 프로젝트 적용 가이드

이 메타 프롬프트는 Invoice MVP 외에도 유사한 SaaS 프로젝트에 적용할 수 있습니다.

### 적용 가능한 프로젝트 유형

- 외부 데이터 소스(Notion, Airtable, Google Sheets)를 읽어 공개 페이지로 제공하는 서비스
- 관리자와 일반 사용자(클라이언트/고객)의 역할이 분리된 서비스
- 문서/자료를 생성하여 고유 링크로 공유하는 서비스
- PDF 생성/다운로드 기능이 포함된 서비스

### 커스터마이징 포인트

프롬프트 지시문에서 다음 항목을 교체합니다.

1. "프로젝트명": 실제 프로젝트 이름으로 교체
2. "핵심 문제": 해결하려는 문제를 구체적으로 교체
3. "사용자 유형": 실제 사용자 역할을 나열
4. "기술 스택": 실제 사용하는 기술과 버전으로 교체
5. "외부 연동": 연동하는 외부 서비스와 환경 변수 교체

### 외부 연동 서비스별 고려사항

| 연동 서비스 | 데이터 소스 특성 | 동기화 전략 |
|-------------|------------------|-------------|
| Notion API | 페이지/DB 기반, API 호출 한도 있음 | 수동 동기화 버튼 + DB 캐싱 권장 |
| Airtable API | 레코드 기반, 웹훅 지원 | 웹훅으로 자동 동기화 가능 |
| Google Sheets API | 스프레드시트 기반, 실시간성 낮음 | 주기적 폴링 또는 수동 동기화 |
| Slack API | 이벤트 기반, 실시간 알림 | 웹훅 이벤트 수신 |
