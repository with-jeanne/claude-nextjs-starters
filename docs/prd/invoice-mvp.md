# Invoice MVP PRD

## 핵심 정보

**목적**: Notion에 입력한 견적서 데이터를 클라이언트가 고유 링크로 웹에서 확인하고 PDF로 다운로드할 수 있게 하여, 이메일 첨부 방식의 비효율을 제거한다.

**사용자**: 프리랜서/소규모 사업자(관리자)와 견적서를 받는 클라이언트(수신자)

---

## 사용자 여정

```
1. [관리자] Notion 데이터베이스에 견적서 입력
   ↓ 저장 완료 후 대시보드 접근

2. 대시보드 페이지 (관리자 전용)
   ↓ 로그인 여부 확인

   [비로그인] → 로그인 페이지 → 로그인 성공 → 대시보드 페이지
   [로그인됨] → 대시보드 페이지 (바로 진입)
   ↓

3. 대시보드 페이지
   ↓ 특정 견적서 선택 → 공개 링크 복사 또는 견적서 상세 확인
   ↓

4. [클라이언트] 공개 링크 접근 (비로그인)
   ↓ 링크 유효성 확인

   [유효한 링크] → 견적서 공개 페이지
   [만료/없는 링크] → 오류 페이지
   ↓

5. 견적서 공개 페이지
   ↓ PDF 다운로드 버튼 클릭

6. PDF 파일 다운로드 완료
```

## 기능 명세

### 1. MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|------------|
| **F001** | Notion 데이터 동기화 | Notion API로 견적서 DB 조회 및 캐싱 | 데이터 소스 연결 없이 서비스 불가 | 대시보드 페이지, 견적서 공개 페이지 |
| **F002** | 견적서 목록 조회 | 동기화된 견적서 목록 표시 및 상태 확인 | 관리자의 핵심 업무 화면 | 대시보드 페이지 |
| **F003** | 공개 링크 생성 | 견적서별 고유 토큰 기반 공개 URL 생성 | 클라이언트에게 견적서 전달 수단 | 대시보드 페이지 |
| **F004** | 견적서 공개 페이지 | 토큰으로 견적서 내용 웹 렌더링 | 클라이언트의 핵심 소비 화면 | 견적서 공개 페이지 |
| **F005** | PDF 다운로드 | 공개 페이지 내용을 PDF로 변환 및 저장 | 클라이언트 최종 수령 수단 | 견적서 공개 페이지 |

### 2. MVP 필수 지원 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|------------|
| **F010** | 기본 인증 | 관리자 회원가입/로그인/로그아웃 | 대시보드 보호를 위한 최소 인증 | 로그인 페이지, 회원가입 페이지 |
| **F011** | 견적서 상태 관리 | 견적서 상태값 업데이트 (발송됨/확인됨 등) | 관리자 업무 추적의 핵심 | 대시보드 페이지 |

### 3. MVP 이후 기능 (제외)

- 프로필 상세 관리 (로고, 서명 이미지 등)
- 견적서 템플릿 커스터마이징
- 클라이언트 서명/승인 기능
- 이메일 자동 발송
- 견적서 만료일 설정
- 실시간 열람 알림

```
공개 영역 (비로그인 접근 가능)
├── 견적서 공개 페이지 (고유 토큰으로 접근)

```

## 데이터 모델

### Invoice (견적서)

| 필드 | 설명 | 타입/관계 |
|------|------|-----------|
| id | 고유 식별자 | UUID |
| notion_page_id | Notion 페이지 ID (동기화 키) | String |
| title | 견적서 제목 | String |
| client_name | 수신 클라이언트명 | String |
| total_amount | 합계 금액 | Number |
| status | 상태 (draft / sent / confirmed) | Enum |
| public_token | 공개 링크용 고유 토큰 | String |
| issued_at | 발행일 | Date |
| synced_at | 마지막 Notion 동기화 시각 | Date |

### InvoiceItem (견적 항목)

| 필드 | 설명 | 타입/관계 |
|------|------|-----------|
| id | 고유 식별자 | UUID |
| invoice_id | 소속 견적서 | → Invoice.id |
| description | 항목 설명 | String |
| quantity | 수량 | Number |
| unit_price | 단가 | Number |
| amount | 소계 (quantity x unit_price) | Number |

### User (관리자)

| 필드 | 설명 | 타입/관계 |
|------|------|-----------|
| id | 고유 식별자 | UUID |
| email | 로그인 이메일 | String |
| created_at | 가입일 | Date |

## 기술 스택

### 프론트엔드 프레임워크

- **Next.js 16.2.4** (App Router) - 현재 프로젝트 기반, 서버 컴포넌트로 Notion API 호출
- **React 19.2.4** - 현재 프로젝트 기반
- **TypeScript 5.6+** - strict 모드

### 스타일링 & UI

- **TailwindCSS v4** (CSS-first, 설정 파일 없음) - 현재 프로젝트 기반
- **shadcn/ui** - 테이블, 모달, 드롭다운, 버튼 컴포넌트
- **Lucide React** - 아이콘

### 폼 & 검증

- **React Hook Form 7.x** - 로그인/회원가입 폼
- **Zod** - 이메일/비밀번호 유효성 스키마

### Notion 연동

- **@notionhq/client** - Notion API 공식 클라이언트, 서버 사이드에서만 호출
- 환경 변수: `NOTION_API_KEY`, `NOTION_DATABASE_ID`

### PDF 생성

- **@react-pdf/renderer** - React 컴포넌트를 PDF로 서버 사이드 렌더링 (권장)
- 대안: `window.print()` + 인쇄 최적화 CSS (구현 단순, 디자인 제한 있음)

### 백엔드 & 데이터베이스

- **Supabase** - 인증(Auth), PostgreSQL 데이터베이스, 서버리스 환경
- **PostgreSQL** - Invoice, InvoiceItem, User 테이블 저장

### 배포 & 호스팅

- **Vercel** - Next.js 최적화, 환경 변수 관리

## 주요 기술 결정 사항

**Notion 동기화 방식**
- 클라이언트가 공개 페이지 접근 시 Notion API를 매번 호출하면 응답이 느리고 API 한도 소진 위험이 있음
- 대시보드에서 수동 동기화 버튼으로 Supabase에 스냅샷 저장, 공개 페이지는 Supabase 데이터를 우선 조회하는 방식 권장

**공개 링크 보안**
- UUID v4 기반 `public_token` 사용 (추측 불가)
- 토큰은 URL에 포함: `/invoice/[token]`
- 별도 만료 처리는 MVP 이후로 연기

**PDF 생성 전략**
- MVP 1단계: `window.print()` + 인쇄 CSS로 빠르게 구현
- MVP 이후: `@react-pdf/renderer`로 서버 사이드 PDF 생성 및 파일 다운로드로 개선
